// case study — LIS 4934

Managing a Software Database
for an Educational Institution

A project implementation plan for a comprehensive software-tracking database at the University of South Florida, guided by Baars's (2006) six-phase project management framework.

Author Mason Good
Course LIS 4934
Date June 7, 2026
Institution University of South Florida
Project Management Database Design ITSM PostgreSQL Software Asset Management Active Directory

Colleges and universities grow increasingly dependent on software across every academic and administrative function — the need for systematic software asset management has never been more critical. From Office 365 to virtual machines, as technology continues to expand so does its use in higher education. With the continued adoption of new software, institutions need a real solution to managing it all.

A comprehensive software-tracking database enables an institution's IT department to monitor software types, vendors, versions, licensing agreements, departmental usage, and individual installation records. That's all essential information for compliance, cost management, and security. The following plan outlines the development of such a database for the University of South Florida (USF) — a large public institution with approximately 50,000 students and 16,000 employees across campuses in Tampa, St. Petersburg, and Sarasota-Manatee. The database will be built and maintained by USF's central IT department, guided by the six-phase project management model described in Baars (2006).

// the six-phase framework

Implementation Plan.

// phase 01
Initiation

The initiation phase can begin only once a CIO has decided the institution needs a solution to its software management issues. Typically, a project manager is assigned to develop and map out a project proposal addressing the project's necessity, feasibility, stakeholder involvement, and scope.

Key activities include not just drafting the proposal, but ensuring all required parties are involved: legal, procurement, finance, IT leadership, and staff. The proposal must be approved by all parties against five key factors, according to Baars (2006): time, money, quality, organization, and information — all to ensure a detailed plan going into the definition phase.

// phase 02
Definition

The definition phase allows the team to review the project proposal one last time and make necessary adjustments before moving to design. As Baars (2006) notes, requirements overlooked at this stage become costly to implement later.

To ensure all needs are captured, the PM will lead structured meetings with IT admins, various USF departments, and end users (students and faculty). These sessions document requirements across four key categories: preconditions, functional requirements, operational requirements, and design limitations. A major goal here is also identifying and resolving any conflicting requirements between stakeholder groups before they become a bigger problem downstream. The final deliverable is an approved requirements document that establishes the contractual baseline for the rest of the project (Baars, 2006).

// phase 03
Design

With an approved proposal and all requirements gathered, the design phase begins. At this point the team evaluates third-party software management tools that match the criteria — options could include an ITSM solution like HaloITSM, licensing something like Azure DevOps, or building a custom solution from scratch.

To build internally, the most practical option would be PostgreSQL, given its JSON support and widespread adoption across modern software infrastructure. Based on the requirements gathered, the database gets built out to house all necessary information: software names, licenses, expiration/renewal dates, assigned users, and more. This would likely require several relational tables. A UI/UX designer would then build a front-end to interact with the database, all hosted internally to eliminate data privacy concerns.

// phase 04
Development

The development phase prepares all people, tools, and infrastructure needed before building begins. Activities include provisioning the development server, establishing version control, finalizing the work breakdown structure and project plan, and confirming team assignments — two database developers, one front-end developer, one QA analyst, and the project manager.

Baars (2006) emphasizes confirming availability commitments from all team members in advance to prevent resource conflicts common in matrix-structured organizations. Project management tooling like Microsoft Project or Smartsheet gives the team a living issue log, action-and-decision list, risk log, and milestone tracking that keeps everything tangible and accountable.

// phase 05
Implementation

Implementation is the build phase. Developers construct the SQL database schema, followed by front-end development of the IT portal, data migration from existing spreadsheets, and Active Directory integration. The QA analyst conducts unit and integration testing against the approved requirements — including license expiration reporting, multi-department queries, and concurrent user load testing.

The project manager tracks progress against the WBS using the five control factors and documents any scope deviations before authorizing changes (Baars, 2006). The phase closes with a formal user acceptance test (UAT) with representative IT staff. The deliverable is a tested, fully functional database that meets all approved requirements.

// phase 06
Follow-Up

The follow-up phase formally closes the project. Activities include producing a system administration guide and user manual, conducting hands-on training for IT staff, establishing a 90-day post-launch support process, and transferring system ownership to the IT Director.

A final project report evaluates outcomes against the original control factors. Baars (2006) cautions that follow-up boundaries must be defined clearly upfront to prevent ongoing operational requests from being treated as project work. At 90 days post-launch, responsibility transfers fully to IT operations and future changes are handled through USF's standard change management process.

// takeaway

Why This Framework Works.

Applying Baars's (2006) six-phase framework to USF's software-tracking database project provides clear structure, formal decision points, and defined deliverables at every stage. This approach directly addresses the failures most commonly experienced in IT projects: scope expansion, vague requirements, and neglected closure — all things I've experienced firsthand.

// personal context

This isn't purely academic for me. In my day-to-day work building ITSM solutions for enterprise clients, I see exactly what happens when a project skips the definition phase or never properly closes out. The Baars framework maps closely to how I already think about implementation work — you can't build the right thing if you haven't nailed down what "right" means before you start.
